# Enometa Shopping Mall — 개발 일지

> 주말 미션으로 동작하는 쇼핑몰을 만들어가는 과정 기록.
> "결과물"뿐 아니라 "어떻게 만들었는지"를 보여주기 위한 문서.

### 이 문서의 원칙
- **전문 용어에는 괄호로 뜻을 함께 표기한다.** 이 프로젝트는 학습 과정이기도 하므로, 처음 보는 사람도 이해할 수 있게 작성한다.
- 같은 용어가 반복 등장할 경우, 첫 등장 시에만 설명을 붙인다.

## Day 0 — 사전 준비 (2026-04-04 금요일)

### 미션 수령

강의에서 주말 미션이 떨어졌다:

> "BMAD method 사용해서 쇼핑몰 하나 만들어보실래요 주말동안에?
> 동작하는 쇼핑몰 만드셔야해요. 모양만 만드는거 아니구요."

단순 과제가 아니라, 이걸 제대로 해서 열정을 어필하고 싶었다. 목표를 세웠다:
- 실제 동작하는 풀스택(frontend+backend를 모두 갖춘) 쇼핑몰
- 체계적 과정 문서화 (BMAD 프레임워크)
- 하이엔드 패션 브랜드 감성의 디자인
- 배포까지 완료

### 레퍼런스 리서치

제일 먼저 한 건 레퍼런스 사이트 조사. **le17septembre.com**을 타겟으로 잡았다 — 미니멀 럭셔리 컨템포러리 패션 브랜드.

레퍼런스 사이트의 레이아웃(화면 배치 구조) 구조를 분석할 때 **Claude 크롬 확장**을 사용했다. 사이트에 접속한 상태에서 Claude에게 UI(User Interface, 사용자가 보는 화면) 구조를 분석하게 시켰다:
- 헤더(상단 고정 메뉴바) 구조 (3칼럼 그리드: 카테고리 / 로고 / LOG IN + CART)
- 상품 리스트 레이아웃 (그리드 간격, 카드 구조)
- 상품 상세 페이지 (이미지 갤러리 + 사이드 패널)
- 전체적인 타이포그래피(글꼴/크기/굵기 체계), 컬러, 간격 패턴

분석 결과를 `레퍼런스/le917분석.md`로 정리했다.

### 레퍼런스 이미지 수집

le17septembre에서 2개 상품의 에디토리얼 이미지를 레퍼런스로 수집:
- WRINKLE KNIT TOP — 9장
- SATIN EASY PANTS — 9장

이 이미지들은 AI 이미지 생성 시 톤앤매너 레퍼런스로 사용할 예정이었다.

### 프로토타이핑 시도 — Google Stitch

**Google Stitch**로 인터랙션 프로토타입을 먼저 만들어보려 했다. 인트로 시퀀스(타이핑 애니메이션 + 카드 등장)와 페이지 전환을 미리 시각화하려는 의도.

**결과: 폐기.**

Stitch에서 원하는 수준의 인터랙션을 만들어내기 어려웠다. 만들어지긴 하는데 마음에 드는 퀄리티가 안 나왔다. 결국 프로토타입 없이 바로 코드로 가기로 결정.

> 교훈: 도구에 시간을 뺏기느니, 바로 코드로 가는 게 나을 때가 있다. 특히 시간이 제한된 주말 스프린트에서는.

### 계획 메모 정리

쇼핑몰에 필요한 것들을 정리한 계획 메모를 작성:
- 사용자 페이지 필수 기능 (상품 전시, 주문/결제, 마이페이지)
- 관리자 페이지 필수 기능 (대시보드, 상품/주문/회원 관리)
- 포트폴리오 전략 (데모 계정, 기술 스택 명시, 문제 해결 사례)
- 디자인 원칙 (Frictionless 결제, Mobile-First, 이미지 최적화)

## Day 1 — BMAD 기획 + 구현 (2026-04-04 금요일 밤 ~ 04-05 새벽)

### BMAD 프레임워크 적용

Claude Code에 BMAD를 설치하고, 체계적으로 기획 문서를 만들어갔다:

**1단계: 브레인스토밍** (`_bmad-output/brainstorming/`)
- Resource Constraints 기법으로 시간 제약 하에서 어디에 집중할지 결정
- SCAMPER 기법으로 le17septembre 대비 차별화 포인트 도출
- 핵심 결정: "인트로 + 상세(95%) > 리스트(90%) > 장바구니(85%) > 관리자(80%)"

**2단계: Product Brief** → **PRD** → **Architecture** → **UX Design** → **Epics & Stories** → **Sprint Plan**

총 7개 문서를 약 2시간에 걸쳐 완성. 각 문서에는 UI 용어 설명 주석을 포함시켰다 (학습 목적).

### 구현 스프린트

BMAD 기획이 끝나자마자 바로 코드 작업에 들어갔다. Epic 1(프로젝트 세팅)부터 Epic 10(마무리)까지 순서대로 진행:

1. **프로젝트 세팅** — Next.js 16 + Supabase 연결 + 타입 정의 + 유틸
2. **공통 레이아웃** — Header(3칼럼), SideMenu, Footer, PageTransition
3. **인트로 페이지** — 타이핑 애니메이션 + 이미지 카드 시퀀스 + ENTER SHOP
4. **상품 리스트** — 카테고리 필터 + 그리드 토글(2/3/4열) + 호버 이미지 스왑
5. **상품 상세** — 이미지 갤러리 + 옵션 선택(사이즈/컬러) + Sticky Bar(화면 하단에 고정되는 구매 버튼)
6. **장바구니 + 체크아웃(결제 진행 페이지)** — 사이드 드로어(옆에서 슬라이드되는 패널) + 배송 정보 폼 + 주문 생성
7. **회원 시스템** — 로그인/회원가입 + 마이페이지 + 미들웨어(요청을 가로채서 권한 체크하는 중간 처리) 보호
8. **문의 시스템** — 플로팅 버튼(화면에 떠있는 고정 버튼) + 모달(팝업 창) 폼
9. **관리자** — 대시보드 + 상품 CRUD(Create/Read/Update/Delete, 생성/조회/수정/삭제) + 주문 관리 + 문의 관리
10. **마무리** — 개인정보/이용약관 + Supabase 시드 데이터(초기 테스트용 샘플 데이터, 12개 상품)

#### 구현 중 겪은 문제들

**문제 1: Supabase RLS(Row Level Security, 행 단위 접근 제어) 정책 설계**
관리자만 상품을 수정할 수 있어야 하는데, RLS 정책의 순서와 조건을 여러 번 수정했다. 결국 `user_metadata.role = 'admin'`(사용자 메타데이터에서 역할이 관리자인지) 체크로 통일.

**문제 2: Next.js 16 미들웨어 경고**
Next.js 16에서 middleware 패턴이 deprecated(더 이상 권장하지 않는, 향후 삭제 예정) 경고를 띄움. "proxy" 패턴으로 마이그레이션(이전) 해야 하지만, 주말 스프린트에서는 동작하는 현재 방식을 유지하기로 결정.

**문제 3: 장바구니 데이터 전략**
PRD에서는 비로그인(localStorage) → 로그인(Supabase) 병합을 계획했지만, 구현 복잡도 대비 데모 효과가 적어 localStorage 단일 전략으로 단순화. cart_items 테이블은 만들어뒀으니 나중에 확장 가능.

### 배포 삽질기

**문제: .git 폴더 분실**

이전 세션에서 프로젝트 구조를 `app/` 하위에서 작업했는데, 새 세션에서 루트에 `.git`이 없었다. `git init` + `git remote add` + `git fetch` + `git reset --mixed origin/main`으로 복구.

**문제: 빈 `app/` 폴더의 유령**

Next.js가 루트의 빈 `app/` 폴더를 App Router(Next.js의 페이지 라우팅 시스템) 디렉토리로 인식해서, 실제 코드가 있는 `src/app/`을 무시했다. 결과: 모든 라우트(URL 경로) 404(페이지 없음 에러).

빌드 출력에서 `Route (app)` 섹션이 아예 안 나오고 `Route (pages) - /404`만 표시되는 걸 보고 원인을 추적. 빈 `app/` 디렉토리 삭제 후 18개 라우트 전부 정상 빌드.

> 교훈: Next.js는 `app/` > `src/app/` 우선순위로 App Router를 감지한다. 루트에 빈 `app/` 폴더가 있으면 전체 라우팅이 죽는다.

**Vercel 배포 성공**

Vercel CLI로 환경변수(Supabase URL, Keys) 설정 후 `vercel --prod`로 프로덕션 배포.

배포 URL: **https://enometa-shopingmall.vercel.app**

### UX 개선: 헤더에 LOG IN 추가

le17septembre 레퍼런스를 다시 보니, 헤더 우측에 LOG IN이 노출되어 있었다. 우리 프로젝트는 사이드 메뉴 안에만 로그인이 있어서 접근성이 떨어졌다.

수정:
- 비로그인 상태: **LOG IN** 표시 (→ /auth/login)
- 로그인 상태: **MY PAGE** 표시 (→ /mypage)
- Supabase `onAuthStateChange`(로그인/로그아웃 이벤트 감지 함수)로 실시간 상태 감지
- 모바일에서는 숨김 (사이드메뉴로 접근)

카테고리 필터 옆에 **LOOKBOOK** 링크도 추가.

| **상품 검색** | 카테고리 필터만 | 키워드 검색 (상품명/설명 대상) | PRD(제품 요구사항 문서)에 명시되어 있지만 12개 상품에는 과한 기능 | 하 |
| **소셜 로그인** | 이메일/비밀번호만 | 카카오/네이버/구글 OAuth(외부 로그인 연동) | Supabase Auth에서 지원하지만 각 플랫폼 앱 등록에 시간 소요 | 중 |
| **middleware → proxy 마이그레이션** | Next.js middleware (deprecated 경고) | Next.js 16 proxy 패턴 | 동작에 문제 없음, 경고만 뜸. 공식 가이드 확인 후 전환 | 하 |
| **커스텀 404 페이지** | Next.js 기본 404 | 브랜드 디자인에 맞는 커스텀 404 | 10분 작업이지만 우선순위 낮음 | 하 |
| **모바일 반응형(화면 크기에 따라 레이아웃 변환) 폴리싱** | 기본 반응형 (1024px 분기점) | 상세 터치 인터랙션(상호작용), 스와이프(밀어서 넘기기) 갤러리, 간격 미세 조정 | 전체적으로 동작하지만 디테일 부족 | 중 |

> **설계 원칙**: 보류한 항목들은 모두 DB 스키마/타입/RLS가 이미 준비되어 있어, 추후 코드만 추가하면 확장 가능하도록 설계했다. "지금 안 만들더라도 나중에 만들 수 있는 구조"를 의식적으로 유지했다.

## 회고

### 잘한 점
- **BMAD 프레임워크 전 과정 적용**: 브레인스토밍 → PRD → 아키텍처 → UX → 에픽/스토리 → 스프린트 → 코드 리뷰 → 배포까지 완주. 과정 문서화가 결과물만큼 중요한 미션에서 큰 강점.
- **Party Mode로 이미지 전략 결정**: AI 에이전트 토론으로 "누끼 포기 → 에디토리얼 통일"이라는 핵심 결정을 내림. 혼자였으면 "다 만들어야지" 하다 시간 부족했을 것.
- **병렬 작업 전략**: 이미지 생성(사용자)과 코드 작업(AI)을 동시 진행. 대기 시간 제로.
- **설계-구현 분리**: 모든 미구현 항목의 DB 스키마/타입/RLS를 미리 설계해둬서, 장바구니 연동 같은 후속 작업이 빠르게 완료됨.

### 아쉬운 점
- **AI 이미지 일관성의 한계**: 같은 프롬프트에서도 옷 디테일이 미묘하게 달라진다. 모델컷과 누끼를 같은 옷으로 만드는 건 현재 기술로 불가능 — 에디토리얼 통일로 우회했지만 실제 쇼핑몰이라면 실물 촬영이 필수.
- **토스페이먼츠 실결제 테스트**: 테스트 키로 위젯은 뜨지만, 실제 카드 결제 → 확인 플로우는 별도 테스트 환경이 필요.

### BMAD 프레임워크 소감
주말 밤샘 스프린트에서 BMAD가 빛난 순간은 **"뭘 먼저 해야 하지?"를 고민하지 않아도 된 것**. PRD와 에픽/스토리가 우선순위를 미리 정해뒀고, 시간 부족 시 "자를 수 있는 것" 목록도 있었다. Party Mode는 의사결정 도구로 효과적이었다 — 여러 관점에서 동시에 검토하니 단독으로 결정했을 때보다 더 나은 결론이 나왔다.
