## Day 3 — 기능 완성 + 이미지 (2026-04-05 일요일 오후~밤)

### BMAD Party Mode: 이미지 전략 회의

AI 에이전트 4명(UX, PM, Architect, Dev)이 토론. 핵심 결정:
- **누끼 포기** → AI로 모델컷과 동일한 옷의 누끼를 일관되게 만들 수 없음. 전 상품 에디토리얼 톤 통일.
- **썸네일 + 호버** = Midjourney 4장 그리드에서 반신(메인) + 전신(호버) 선택
- **톤앤매너** = Kodak Portra 400 필름 톤, 미세 그레인, 자연광, 무심한 포즈

### AI 이미지 생성

Midjourney v7로 **28장** 생성 (12상품 × 2장 + Lookbook 4장). 코드 작업과 **병렬 진행**.

**배운 점:**
- 같은 옷 일관성 → 같은 프롬프트의 4장 그리드 안에서만 선택
- 반신/전신 구분 → `waist-up close-up portrait` 같은 명시적 키워드 필요
- 하의 상품은 `hip-down cropped shot`으로 하체 중심 크롭

### 기능 구현

- **장바구니 Supabase 연동**: 로그인 → Supabase 카트, 비로그인 → localStorage
- **토스페이먼츠**: Payment Widget SDK v1 연동 (테스트 모드)
- **Lookbook 페이지**: 풀와이드 히어로 + 2/4컬럼 그리드 + 스크롤 FadeIn
- **상세 페이지**: 이미지 클릭 확대 모달, 간격 개선
- **모바일 대응**: 카테고리 필터 → 메뉴 안으로, 인트로 카드 5장 한 줄
- **기타**: SEO 메타태그, 가입 후 바로 로그인, 사이드메뉴 로그아웃 상태 반영

### 삽질 기록

**1. 장바구니 수량 무한 증가 버그**
새로고침할 때마다 수량이 2배씩 증가 (1→204→3264). 원인: `onAuthStateChange`가 매번 발생 → localStorage 카트를 Supabase에 반복 병합. 해결: 병합 로직 제거, 로그인 시 Supabase만 사용.
> 교훈: 병합 로직은 멱등성(여러 번 해도 결과 동일)을 반드시 보장해야 한다.

**2. 토스페이먼츠 키 인증 오류**
"인증되지 않은 키" 에러. 원인: `echo` 명령어가 키 끝에 줄바꿈을 추가 + Vercel 빌드 캐시가 새 환경변수를 반영 안 함. 해결: `printf`로 재설정 + `--force` 강제 재배포.
> 교훈: CLI로 환경변수 넣을 때 `echo` 대신 `printf`, 변경 후 캐시 없이 재배포.

**3. Vercel Git 미연결**
push해도 자동 배포 안 됨. 원인: Git 연결이 끊어져 있었음. 해결: CLI `vercel --prod`로 수동 배포 후 Dashboard에서 Git 재연결.

**4. Hydration 에러**
서버 CART(0) vs 클라이언트 CART(4) 불일치. 원인: Zustand persist가 localStorage를 SSR에서 읽지 못함. 해결: `mounted` 상태 추가, 마운트 전에는 0 표시.

### 마무리 — 품질 완성 + 보안 강화 (저녁)

BMAD quick-dev 스타일로 남은 높은 우선순위 4개를 자동 순회. 빌드 검증 루프 포함.

**1. Server Action 보안 강화**
- `src/actions/admin.ts`의 8개 함수에 `requireAdmin()` 권한 체크 추가
- 기존에는 Middleware(경로 보호)만 있었는데, Server Action은 POST 요청으로 직접 호출 가능한 보안 취약점(vulnerability) 존재
- 3중 보호 구조 완성: Middleware → Layout → Server Action
- `docs/admin-설정-가이드.md` 작성 (Supabase에서 admin 설정하는 방법)

**2. 에디토리얼 페이지 (`/editorial`)**
- b컷(behind-the-scenes) 이미지 26장을 활용한 브랜드 스토리텔링 페이지
- Lookbook 페이지의 `FadeInSection`, `LookbookImage` 패턴 재활용
- 풀와이드 히어로 + 2/3컬럼 그리드 + 텍스트 인터미션 구성
- SideMenu에 EDITORIAL 네비게이션 추가

**3. 모바일 반응형 폴리싱**
- 상품 상세 페이지 스와이프 인디케이터 활성화 (IntersectionObserver 기반)
- 현재 보이는 이미지에 맞춰 dot 색상 변경 (`bg-border` → `bg-dark`)
- scroll-smooth 추가, 인디케이터 간격 타이트하게 조정

**4. 낮은 우선순위 8개 일괄 완료**

BMAD 자동 실행으로 남은 TODO 전부 처리:
- **상품 검색**: `/shop` 페이지에 실시간 검색 바 추가 (이름/설명 필터링)
- **재고 검증**: 주문 생성 시 서버에서 재고 확인 후 부족하면 에러 반환
- **주문 상세**: 수령인, 연락처, 상세주소, 배송메모까지 표시하도록 보강
- **관리자 폼 UX**: JSON 직접 입력 → 컬러 피커, 사이즈 토글 버튼, 재고 사이즈별 입력, 이미지 URL 동적 추가/삭제
- **middleware → proxy**: Next.js 16에서 middleware가 deprecated → proxy.ts로 마이그레이션
- **쿠폰 시스템**: 10% 환영쿠폰 (가입 시 자동 발급, 30일 유효, 5만원 이상 주문 시 사용)
- **소셜 로그인**: Google/카카오 버튼 + OAuth 콜백 라우트 (Supabase Provider 설정 필요)
- **Storage**: 설정 가이드 문서 (버킷 생성 + RLS 정책)

### 기술 포인트

- **IntersectionObserver**: 스크롤 위치에 따라 현재 보이는 요소를 감지하는 브라우저 API. 스와이프 갤러리에서 "지금 몇 번째 이미지가 보이는지" 추적하는 데 사용.
- **Server Action 보안**: Next.js Server Action은 URL 기반 미들웨어를 우회할 수 있으므로, 함수 내부에서도 권한을 직접 검증해야 한다.
- **proxy (Next.js 16)**: middleware가 deprecated되고 proxy로 이름이 변경됨. 함수명 `middleware` → `proxy`, 파일명 `middleware.ts` → `proxy.ts`.

### Lighthouse 성능 점검

Lighthouse 모바일 측정 결과:
- Performance **65점** / Accessibility **93점** / Best Practices **100점** / SEO **100점**
- LCP(Largest Contentful Paint, 화면에 가장 큰 요소가 보이는 시점)가 17.8초로 느린 게 주요 원인

**개선 작업:**
1. 전체 `<img>` 태그를 `next/image`의 `Image` 컴포넌트로 전환 (13곳)
   - 자동 AVIF/WebP 변환 (PNG 2MB → WebP ~300KB 수준)
   - `sizes` 속성으로 뷰포트별 적정 크기 요청
   - `priority` 속성으로 히어로 이미지 + 첫 화면 상품 4개 즉시 로드
2. `next.config.ts`에 `formats: ['image/avif', 'image/webp']` 추가
3. 인트로 페이지 이미지도 전환 (`onError` 핸들러 → state 기반으로 변경)
4. 커스텀 404 페이지 추가 (브랜드 디자인 통일)

**개선 결과 (/shop 페이지 기준):**
- Performance **75점** / Accessibility **95점** / Best Practices **100점** / SEO **100점**
- Speed Index 5.2초 → **2.8초** (Good), LCP 51초 → **15초**

**80점을 못 넘긴 이유 (의도적 트레이드오프):**
- **이미지 리사이즈 포기**: 상세 페이지에 이미지 확대(줌) 기능이 있어서 원본 해상도를 유지해야 함. 리사이즈하면 줌했을 때 깨진다.
- **인트로 애니메이션 유지**: Framer Motion으로 2~3초 delay를 주는 연출이 브랜드 감성의 핵심. 줄이면 점수는 오르지만 인상이 약해진다.
- **결론**: 패션 쇼핑몰에서 이미지 품질과 연출 감성은 성능 점수보다 중요하다. 75점은 수용 가능한 균형점.

> 교훈: Lighthouse 점수는 "높을수록 좋다"가 아니라 "어디서 타협할지 결정하는 도구"다. `<img>` → `next/image` 전환만으로 할 수 있는 건 했고, 나머지는 디자인 의사결정의 영역.

## 회고

### 잘한 점
- **BMAD 전 과정 완주**: 기획 → 구현 → 배포까지 문서화. 과정이 곧 결과물.
- **Party Mode 의사결정**: "누끼 포기 → 에디토리얼 통일" 결정이 시간을 아끼면서 브랜드 감성을 살림.
- **병렬 작업**: 이미지 생성과 코드 작업을 동시 진행. 대기 시간 제로.
- **확장 가능 설계**: 미구현 항목도 DB 스키마/타입이 준비되어 코드만 추가하면 됨.

### 아쉬운 점
- **장바구니 병합 설계 미스**: 멱등성을 고려 안 해서 수량 무한 증가. "이 함수가 여러 번 호출되면?"을 먼저 물었어야 했다.
- **환경변수 삽질**: `echo` 줄바꿈 + Git 미연결 + 빌드 캐시가 겹쳐서 토스 연동에 시간 소모. 인프라는 작아 보여도 디버깅이 어렵다.
- **AI 이미지 한계**: 같은 옷의 누끼를 일관되게 만들 수 없다. 실제 쇼핑몰이라면 실물 촬영 필수.

### BMAD 소감
BMAD가 빛난 순간은 **"뭘 먼저 해야 하지?"를 고민하지 않아도 된 것**. PRD와 에픽/스토리가 우선순위를 정해뒀고, 시간 부족 시 "자를 수 있는 것" 목록도 있었다. Party Mode는 여러 관점에서 동시 검토 → 더 나은 결론.
